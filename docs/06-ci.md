# CI environment notes

> Authoritative description of what GitHub Actions runs for LDB, what
> tests SKIP on a default Ubuntu runner, and how to reproduce the CI
> environment locally. The workflow itself is `.github/workflows/ci.yml`;
> this file documents the *why*. Updated when the workflow changes.

## What CI runs

Every push to `master` and every pull request against `master` triggers
the `linux` job on a clean `ubuntu-24.04` runner:

1. `actions/checkout@v4`.
2. apt-install: `ninja-build cmake build-essential liblldb-dev lldb
   libsqlite3-dev zlib1g-dev python3 bpftrace tcpdump openssh-server
   openssh-client`.
3. `sudo sysctl -w kernel.yama.ptrace_scope=0` so `target.attach`-style
   tests work against non-child processes.
4. Generate an ed25519 SSH key, install it as an `authorized_keys`
   entry, start `sshd`, scan `localhost` into `known_hosts`. This
   unlocks `target.connect_remote_ssh` smoke coverage that would
   otherwise SKIP.
5. `cmake -B build -G Ninja -DLDB_LLDB_ROOT=/usr/lib/llvm-18
   -DCMAKE_PREFIX_PATH=/usr`.
6. `cmake --build build --parallel`.
7. `build/bin/ldbd --version` (smoke).
8. `ctest --test-dir build --output-on-failure`.
9. On failure only: upload `build/Testing/`, `CMakeOutput.log`,
   `CMakeError.log`, `.ninja_log` as the `linux-failure-logs`
   artifact.

A second job `release` is **gated on tags matching `v*.*`** and
**depends on `linux`**. It does a Release-mode build of `ldbd`,
tarballs the binary as `ldbd-<tag>-linux-x86_64.tar.gz`, and uploads
that as a workflow artifact (90-day retention). Operator policy
decides whether to promote the artifact to a GitHub Release; the
workflow does not auto-create one.

The job has a 30-minute timeout. A green run on the dev box
historically completes in ~30 seconds; the budget allows for runner
cold-cache, apt mirror flakiness, and any `[live]` test that needs to
spawn a subprocess.

## What SKIPs on a default Ubuntu runner

Several `[live]` tests are written to SKIP gracefully when the
required capability or service isn't present. The runner does not
have CAP_BPF, CAP_NET_RAW, or root by default, so:

| Test | Status on GHA runner | Why |
|---|---|---|
| `smoke_uprobe_bpf` | partial (catalog + discovery branch) | The "bpftrace not installed" path runs; the live happy path SKIPs because the runner is non-root and lacks CAP_BPF. |
| `smoke_observer_tcpdump` | partial (catalog + arg validation) | The capture branch SKIPs because the runner lacks CAP_NET_RAW for `tcpdump -i lo`. |
| `smoke_connect_remote_ssh` | full (positive path active) | `sshd` is started and key auth is wired up before ctest, so the positive path is exercised. |
| `smoke_attach`, `smoke_live_*` | full | `kernel.yama.ptrace_scope=0` is set before ctest. |

If a future test needs CAP_BPF/CAP_NET_RAW unconditionally, it should
follow the existing pattern and SKIP cleanly when the capability is
missing â€” failing the matrix on a runner-policy issue is not useful
signal. Tests that are environmentally dependent should print a clear
"smoke: <thing> not available; SKIP" line so the ctest log explains
itself.

`bpftrace` and `tcpdump` are still installed even though their live
paths SKIP â€” having the binaries on PATH means the discovery branches
of those tests exercise code instead of short-circuiting on
`which: not found`.

## What the matrix does NOT cover (deferred)

- **macOS arm64.** Tracked as Tier 1 Â§2 in
  `docs/POST-V0.1-PROGRESS.md`; gated on B1 (the dlopener fixture
  build break) and hardware sign-off. Not a single-line YAML add â€” the
  smoke suite has Linux-flavored skips that need broadening first.
- **clang vs gcc.** Single-compiler matrix (system gcc-13). clang on
  the same OS is covered locally and adds runtime without surfacing
  bugs that gcc misses.
- **Debug vs Release.** ctest runs the default config (Debug). The
  release job builds `Release` for the artifact upload but does not
  re-run the suite â€” the determinism gate is shape-based and unaffected
  by `-O2`.
- **Cross-arch (aarch64 Linux, riscv64).** Tier 6 work; out of scope
  for v0.1 polish.
- **Coverage / sanitizers.** Defer until a workflow needs them.
- **Auto-bump LLDB pin.** Roadmap Â§6.3 calls for a weekly LLDB-pin
  bump bot. Defer until the first compatibility break shows up.

## Reproducing the CI environment locally

On a Linux dev box that closely matches the runner (Ubuntu 24.04 LTS):

```bash
sudo apt-get update
sudo apt-get install -y \
  ninja-build cmake build-essential \
  liblldb-dev lldb \
  libsqlite3-dev zlib1g-dev \
  python3 \
  bpftrace tcpdump

sudo sysctl -w kernel.yama.ptrace_scope=0   # session-only

cmake -B build -G Ninja \
  -DLDB_LLDB_ROOT=/usr/lib/llvm-18 \
  -DCMAKE_PREFIX_PATH=/usr
cmake --build build --parallel
ctest --test-dir build --output-on-failure
```

If you want the SSH-positive path:

```bash
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519 -N ""   # if you don't have one
cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
ssh-keyscan -t ed25519 localhost >> ~/.ssh/known_hosts
sudo systemctl start ssh
ssh -o BatchMode=yes localhost true   # smoke
```

### Why apt LLDB 18 in CI but `/opt/llvm-22` (or `brew llvm`) locally?

CI optimizes for **fast, deterministic, no-network-after-apt** runs.
A prebuilt LLVM 22 tarball is ~700 MB and has to be cached or
downloaded each run; apt's `liblldb-dev` is on the runner image's
mirror and installs in seconds. The SBAPI surfaces LDB uses are
stable across LLVM 18 and LLVM 22 â€” the daemon advertises 18 as the
floor in `README.md` precisely so we can run CI against the apt-shipped
version. Local development against the prebuilt LLVM 22 stays
supported because `LDB_LLDB_ROOT` parameterizes the prefix.

If a future feature needs an LLVM â‰Ą 19 SBAPI, this trade flips and CI
will need to fetch the upstream tarball; document the bump in this
file at the same time.

## When the workflow itself changes

There is a tiny ctest sanity check (`check_ci_yaml`, registered in
`tests/CMakeLists.txt`, sourced in `tests/check_ci_yaml.py`) that
parses `.github/workflows/ci.yml` and asserts the documented shape:
top-level keys, push+pull_request triggers, `v*.*` tag trigger, the
ubuntu-24.04 build job's apt deps and ctest invocation, and the
release job's `ldbd` artifact upload. It is not a full GitHub Actions
linter â€” it exists so a future YAML edit that drops a step shows up
in `ctest -j` instead of as a red CI badge after push.

If you reshape the workflow, run `ctest -R check_ci_yaml` and update
the assertions to match the new shape. The point of the test is to
keep this doc and the YAML in sync.
